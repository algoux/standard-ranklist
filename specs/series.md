# Rank Series

This document defines rank series, segments, and the rule presets used to compute rankings from ranklist data.

## Overview

A **rank series** represents a single ranking dimension within a ranklist. A ranklist MAY contain multiple series, each computing an independent ranking. For example, a contest might have:

- A primary "Overall Rank" series
- A "School Rank" series that picks the top-ranked team from each school
- A "Girls Team Rank" series filtered to specific markers

Each series defines how users are ranked and how segments (medal tiers) are assigned.

---

## RankSeries

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `title` | `string` | No | Generated by renderer | Series title displayed in the ranklist header |
| `segments` | `array<`[RankSeriesSegment](#rankseriesegment)`>` | No | `[]` | Ordered segment definitions (highest tier first) |
| `rule` | [RankSeriesRulePreset](#rule-presets) | No | — | The rule preset used to compute this series |

When `rule` is absent, the series provides visual segmentation only, with no algorithmic rank computation.

---

## RankSeriesSegment

A segment represents a tier within a rank series (e.g., gold medal, silver medal).

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `title` | `string` | No | Ignored by renderer | Segment display title |
| `style` | [Style](data-types.md#style) \| [RankSeriesSegmentStylePreset](#style-presets) | No | Determined by renderer | Visual style for rows in this segment |

Segments are ordered within the `segments` array from **highest tier to lowest tier**. The first segment corresponds to the top-ranked group (e.g., gold medalists), the second to the next tier (e.g., silver medalists), and so on.

> **Segment–Algorithm Alignment:** When a rule preset (e.g., ICPC) uses a `count.value` or `ratio.value` array, each element corresponds to the segment at the same index. If the `value` array is longer than the `segments` array, the extra entries define anonymous segments with no title or style. If the `segments` array is longer than the `value` array, the extra segments receive no users. Implementations SHOULD handle both cases gracefully.

### Style Presets

The `style` field MAY be a preset string instead of a [Style](data-types.md#style) object. The following presets are defined:

| Preset | Intended Meaning |
|--------|-----------------|
| `"gold"` | Gold tier (highest) |
| `"silver"` | Silver tier |
| `"bronze"` | Bronze tier |
| `"iron"` | Iron tier (lowest named tier) |

The actual visual appearance of style presets is determined by the renderer implementation.

### Example

```json
{
  "title": "Gold Medalist",
  "style": "gold"
}
```

## Rule Presets

A rule preset defines the algorithm for computing ranks within a series. Three presets are defined.

### RankSeriesRulePreset Type

**Type:** `RankSeriesRulePresetNormal | RankSeriesRulePresetUniqByUserField | RankSeriesRulePresetICPC`

The `preset` field within the rule object determines which preset is in use.

---

### Preset: Normal

A simple ranking rule that assigns ranks in ascending order based on score.

**Example:**

```json
{ "preset": "Normal" }
```

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `preset` | `"Normal"` | Yes | — | Preset identifier |
| `options` | `object` | No | — | Optional configuration |
| `options.includeOfficialOnly` | `boolean` | No | `false` | If `true`, only official users are ranked |

**Behavior:**

1. Select users: all users, or only official users if `includeOfficialOnly` is `true`.
2. Users are ranked by their score in descending order (higher score = better rank).
3. Users with equal scores receive the same rank.

> **Note:** The Normal preset computes ranks only. It does **not** define a segment assignment algorithm. The `segments` array, if present on the parent `RankSeries`, is used for visual presentation only. To assign users to segments algorithmically, use the [ICPC preset](#preset-icpc) with `count` or `ratio` options.

---

### Preset: UniqByUserField

A ranking rule that deduplicates users by a specified field value, keeping only the highest-ranked user for each unique field value.

**Example:**

```json
{
  "preset": "UniqByUserField",
  "options": {
    "field": "organization"
  }
}
```

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `preset` | `"UniqByUserField"` | Yes | — | Preset identifier |
| `options` | `object` | Yes | — | Configuration (required) |
| `options.field` | `string` | Yes | — | The user field to group by |
| `options.includeOfficialOnly` | `boolean` | No | `false` | If `true`, only official users are ranked |

**Allowed `field` values:**

| Value | Refers to |
|-------|-----------|
| `"id"` | `User.id` |
| `"name"` | `User.name` |
| `"organization"` | `User.organization` |

**Behavior:**

1. Select users: all users, or only official users if `includeOfficialOnly` is `true`.
2. Group users by the specified field value.
3. For each group, keep only the user with the highest rank (best score) from the overall ranklist.
4. Assign ranks to the selected users in ascending order based on their original rank positions.

> **Example:** With `field: "organization"`, if three teams from "MIT" are ranked 2nd, 5th, and 8th overall, only the team ranked 2nd is included in this series.

---

### Preset: ICPC

The most comprehensive rule preset, designed for ICPC-style medal allocation with support for ratio-based and count-based segment sizing, user filtering, and tie-handling options.

> **Note:** The ICPC preset does not provide an `includeOfficialOnly` option. Per ICPC convention, only users with `official` set to `true` (or absent) are considered for segment allocation. Unofficial users MUST be excluded from the denominator and segment assignment calculations.

**Example:**

```json
{
  "preset": "ICPC",
  "options": {
    "count": { "value": [4, 4, 4] }
  }
}
```

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `preset` | `"ICPC"` | Yes | — | Preset identifier |
| `options` | `object` | Yes | — | Configuration (required) |
| `options.ratio` | [RatioAlgorithm](#ratio-algorithm) | No | — | Ratio-based segment sizing |
| `options.count` | [CountAlgorithm](#count-algorithm) | No | — | Count-based segment sizing |
| `options.filter` | [Filter](#filter) | No | — | User filter |

#### Multiple Algorithms

When both `ratio` and `count` are specified, a user is assigned to a segment **only if all specified algorithms agree** on the assignment. This means the final segment boundary is the intersection (minimum) of all algorithm results.

> **Example:** If `ratio` computes gold for ranks 1–25 and `count` computes gold for ranks 1–20, the effective gold segment includes ranks 1–20 only.

#### Ratio Algorithm

Assigns segments based on a ratio of the total user count.

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `value` | `array<number>` | Yes | — | Ratio for each segment (ordered highest to lowest tier) |
| `rounding` | `string` | No | `"ceil"` | Rounding method: `"floor"`, `"ceil"`, or `"round"` |
| `denominator` | `string` | No | `"all"` | How to count the total user number |
| `noTied` | `boolean` | No | `false` | Whether to force no tied ranks for calculation |

**`value` array:**

Each element MUST be a number greater than 0 and less than or equal to 1, representing the proportion of users in that segment. The values are ordered from the highest tier to the lowest tier.

The sum of all ratio values SHOULD NOT exceed 1. If the cumulative segment sizes exceed the total user count, excess segments are empty.

For example, `[0.1, 0.2, 0.3]` means: 10% gold, 20% silver, 30% bronze.

**`rounding` method:**

When the computed segment size is not an integer, the rounding method determines how to round:

| Method | Behavior | Example: 248 users × 10% = 24.8 |
|--------|----------|----------------------------------|
| `"ceil"` | Round up | 25 users |
| `"floor"` | Round down | 24 users |
| `"round"` | Round to nearest | 25 users |

**`denominator` option:**

Specifies how the total user count (the denominator in ratio calculations) is determined. When a [Filter](#filter) is specified, the denominator is always computed **after** filtering — that is, only users passing the filter are counted.

| Value | Meaning |
|-------|---------|
| `"all"` | Total number of filtered users |
| `"submitted"` | Number of filtered users who submitted at least one solution |
| `"scored"` | Number of filtered users whose score value is greater than zero |

**`noTied` option:**

When `true`, tied ranks are forcefully resolved to sequential ranks before computing segment boundaries. For example, if the sorted ranks are `[1, 1, 3, 4]`, they are treated as `[1, 2, 3, 4]` for the purpose of segment calculation. This ensures that the exact number of users specified by the ratio falls into each segment, regardless of ties.

The expansion order MUST follow the `rows` array order: among tied users, the user appearing earlier in the `rows` array receives the higher (numerically lower) rank.

#### Count Algorithm

Assigns segments based on a fixed number of users per segment.

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `value` | `array<integer>` | Yes | — | Fixed size for each segment (ordered highest to lowest tier) |
| `noTied` | `boolean` | No | `false` | Whether to force no tied ranks for calculation |

**`value` array:**

Each element is a non-negative integer specifying the exact number of users in that segment. Values are ordered from the highest tier to the lowest tier.

For example, `[4, 4, 4]` means: 4 gold, 4 silver, 4 bronze.

**`noTied` option:**

Behaves the same as in the [Ratio Algorithm](#ratio-algorithm).

#### Filter

Filters determine which users participate in the segment calculation.

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `byUserFields` | `array<`[UserFieldFilter](#userfieldfilter)`>` | No | — | Filter by user field values |
| `byMarker` | `string` | No | — | Filter by marker ID (exact match) |

All specified filter conditions are combined with logical AND: a user is included only if **all** conditions are satisfied.

##### UserFieldFilter

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `field` | `string` | Yes | — | The user field to filter on |
| `rule` | `string` | Yes | — | A regular expression pattern (used as a `RegExp` constructor string) |

**Allowed `field` values:** `"id"`, `"name"`, `"organization"`

The `rule` string is interpreted as a regular expression using **ECMAScript (JavaScript) RegExp syntax** — equivalent to `new RegExp(rule)`. No flags are applied. The match uses **partial matching** semantics (equivalent to `RegExp.prototype.test()`): a user is included if any substring of their field value matches the pattern. To perform a full-string match, use anchors explicitly (e.g., `"^SDUT$"`).

If the `rule` string is not a valid regular expression, implementations SHOULD treat the filter condition as non-matching (i.e., exclude all users for that condition).

> **Example:** `{ "field": "organization", "rule": "SDUT" }` includes users whose organization contains the substring matching `/SDUT/`, such as `"SDUT"` or `"SDUT-ACM"`.

##### `byMarker`

*Since version 0.3.9.*

When specified, only users that have the given marker ID in their `markers` array (exact string match) are included.

> **Example:** `{ "byMarker": "girls-team" }` includes only users with `"girls-team"` in their `markers` array.

### Segment Assignment Example

Consider a contest with 100 official users and the following ICPC rule:

```json
{
  "preset": "ICPC",
  "options": {
    "ratio": {
      "value": [0.1, 0.2, 0.3],
      "rounding": "ceil",
      "denominator": "all"
    }
  }
}
```

With 3 segments defined (Gold, Silver, Bronze):

1. **Gold** (segment 0): 100 × 0.1 = 10.0 → ceil → **10 users** (ranks 1–10)
2. **Silver** (segment 1): 100 × 0.2 = 20.0 → ceil → **20 users** (ranks 11–30)
3. **Bronze** (segment 2): 100 × 0.3 = 30.0 → ceil → **30 users** (ranks 31–60)

Users ranked 61 and below are not assigned to any segment.
